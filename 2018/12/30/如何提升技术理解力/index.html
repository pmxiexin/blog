如何提升技术理解力

发表于 2018-12-30 | 分类于 经验总结
字数统计: 1.3k | 阅读时长 ≈ 4 分钟

技术层次

  1. 了解技术概念，技术的定义、基本使用方式等
  2. 能正确应用技术于需求中
  3. 能为实现逻辑提出优化建议

了解技术概念

了解程度

  1. 20%的常用技术知识，以应对80%的应用情况
  2. 80%的非常用技术知识，无需刻意学习，每次遇到时快速理解即可

必备的技术知识：

  1. 数据库
    1. 什么是数据库？主要起什么作用？
    2. 有哪些数据库类型？分别有什么特征和应用场景？
    3. 数据库怎么使用？怎么查询数据？
  2. 服务端
    1. 服务端是什么？有什么作用？
    2. 如何处理处理过载的问题？如高并发。
  3. 客户端
    1. 客户端有哪些？分别有什么特征和应用场景？
    2. 客户端和服务端的区别是什么？
    3. 哪些逻辑是客户端处理？哪些逻辑服务端处理
  4. 通信
    1. 通信有哪几种方式？有什么特征和应用场景？（接口/推送）
    2. 什么是接口？接口怎么阅读和使用？

了解技术的内容

  1. 定义：该技术有哪些特征，如何定义
  2. 作用：该技术能起到什么样的作用，实现什么样的功能
  3. 边界：能起到的最大使用效果，什么功能无法实现
  4. 限制：该技术使用中有哪些限制条件
  5. 常用方式：该技术常用的应用逻辑和使用场景
  6. 相似技术：有哪些技术与其特征或功能相似

积累基础技术知识方法（80%应用场景）

  1. 日常工作积累：每个经历过的项目所运用的技术知识都应详细了解
  2. 书籍/文档阅读：产品适合基础类的书籍以丰富知识如《计算机基础》，阅读各个客户端/合作方的开发者文档可以帮助产品理解“通信类知识”
  3. 技术好奇心：时刻保持对技术的好奇心，在使用和体验功能时要经常好奇“这个功能可能是如何实现的？”

快速理解未知的新技术（20%非常规情况）

需要具备以下能力

  1. 沟通能力：能够让开发愿意为你一遍遍解释知识。
    注意让开发根据你的理解回答，如“你看我说的对不对”
  2. 抽象能力：将深奥的开发知识抽象成简单易懂的日常信息，如将“推送”理解为“发短信”
  3. 逻辑能力：将抽象后被理解的产物，按照自己的思维方式去理解，去与开发沟通/校验。
  4. 辩证能力：可确认开发所讲的论据是真实可靠的，技术方案是事实并可行的（不会被忽悠）

技术应用（根据信息的流程）

  1. 信息存储，回答以下问题：
    1. 业务分为哪几个流程？每个流程会产生哪些信息？
    2. 每一类信息分别有哪些具体的字段？
    3. 信息会以什么格式存储（mysql/radis）？
  2. 信息读取
    1. 业务分为哪几个流程？每个流程分别需要读取哪些信息？
    2. 读取信息经过了什么样的筛选/排序/分组方式？
  3. 信息传递
    1. 业务分为哪几个流程？每个流程分别需要传递哪些信息？
    2. 传递的信息以什么样的格式（json/xml……)传递？
    3. 什么样的方式(post/get/rpc……)传递？
    4. 传递后又什么样的反馈？以及反馈后的操作？
  4. 信息展示
    1. 客户端在什么样的情况下需显示信息？
    2. 客户端怎么样获取到信息？
    3. 客户端会有什么样的表达方式（列表、图表……）？
    4. 获取到的信息如何和表达方式进行匹配？
  5. 信息接受
    1. 业务分为哪几个流程？哪些流程会有信息的接受？
    2. 信息接受会有什么来源（用户触发/脚本运行/其他触发）？
    3. 接受信息后会有什么样的操作？以及返回给来源的反馈？

为技术实现提供建议

产品经理无需在技术专业领域提供建议（切忌太懂），仅需做好以下几点

  1. 避免技术冗余：告知开发产品的极限状态和适用情况，避免开发因背景不清晰而造成技术冗余，花大量的时间处理极小概率情况。
  2. 优化产品流程：产品需与技术在功能流程上详细沟通，避免不必要的步骤导致的复杂开发。
  3. 帮助整理类似产品：帮助技术寻找类似的产品，让开发能够定向研究，避免额外的学习和调研成本

技术理解/表达技巧

  1. 业务框架图：用以理解各个业务之间的关系
  2. 流程图：用以理解单一业务的各个过程
  3. 时序图（以过程为核心）：用以理解一个重要过程的信息处理关系
  4. 状态机（以状态为核心）：用以处理一个重要过程的状态变化关系
  5. 信息流图（以信息传递为核心）：用以处理一个重要过程的信息传递关系
