历史项目风险整理

发表于 2019-02-19 | 分类于 经验总结
字数统计: 1k 字 | 阅读时长 ≈ 5 分钟

产品Lead风险

  1. 项目立项过于草率，缺少数据支持，后期推翻重做
  2. 项目阶段性规划不清晰，方向频繁变动
  3. 项目立项准备过长，反复犹豫，延误最佳上线时机

产品经理风险

  1. 需求文档描述不清晰，存在歧义
  2. 需求文档考虑不全面，如存在逻辑漏洞，如未与相关同事确认等，未经老板审批等，频繁进行需求变更
  3. 需求撰写时间过长，延误最佳上线时机
  4. 需求未完全确认，部分设计、开发没有提出意见，后期才发现。导致需求变更
  5. 需求变更未周知所有项目人员，导致理解偏差，进行返工
  6. 需求变更文档未经整理，导致后期难以维护
  7. 产品没有准备好上线必须的条件，如第三方工具的账号申请等
  8. 上线后未进行相关人员的周知，导致上线后无人跟进问题

项目经理风险

  1. 没有安排足够的人员，或是安排合适的人员进行参与，导致人员工作效率低于预期
  2. 排期估时不严谨或是考虑不全面，比如没有考虑到历史功能的影响、多个版本的兼容性、熟悉代码的时间、新人准备的时间等，有人估多有人估少，导致排期计划频繁变动
  3. 排期考虑不全面，如未考虑节假日、未考虑人员加班、未考虑缓冲时间等，导致排期频繁变动
  4. 没有进行合理的团队激励，调和内部关系，导致团队工作效率低于预期
  5. 当计划进行变动时，如提前或延期上线，没有立即进行响应
  6. 缺少进度的及时跟踪，导致进度异常无法被及时纠正处理
  7. 不重视流程，导致各个阶段的交接都不符合规范，造成沟通偏差和工作失误
  8. 太过重视流程，导致团队需投入过多精力应付流程，造成效率低下

设计师风险

  1. 设计未阅读产品文档，以及于产品沟通详细需求，最终设计稿不符合要求
  2. 设计未按照既定的设计规范和设计风格，频繁的进行返工
  3. 设计未按照指定要求的格式进行输出，在输出阶段进行返工

开发者风险

  1. 开发未阅读产品文档，以及于产品沟通详细需求，最终开发结果不符合要求
  2. 开发采用了错误的实现方式，或是进入了不熟悉的领域，频繁地进行返工
  3. 开发擅自修改需求，导致与要求不符，进行返工
  4. 出现问题时没有进行及时汇报，等待他人发现，导致项目延期
  5. 多人进行合作的开发项目，缺少前期的沟通准备，导致合并代码时出现错误
  6. 没有进行自测，导致产出的代码需进行返工

测试风险

  1. 测试未阅读产品文档，以及于产品沟通详细需求，没有测试全面
  2. 测试所反馈的Bug描述不清晰，降低工作效率
  3. 缺少上线前公开的的体验和细节调优，导致上线后效果不满意
  4. 上线时未按照正确的上线流程，导致上线后出现bug，需多次上线

客观问题

  1. 依赖于他人的部分没有如期完工，如外包技术、技术合作等
  2. 第三方技术的不稳定性，出现了无法预测的问题
  3. 不可抗力的政策风险、平台规范的变化
  4. 上线时要求的设施没有到位，或是不符合要求

所以怎么样能解决这些问题呢？我的答案是“文化和流程”
